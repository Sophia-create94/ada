Project overview

Build a two-page portfolio website for Ada — an AI-powered accommodation booking app. Ada is a smart meta-search tool that finds five curated stays matching a user's taste, instead of showing hundreds of results. The two pages are a landing page and a results page. Three reference files are in the project root — read all of them before writing any code:

  • ada-landing.html — full landing page: copy, layout, interactions, design tokens
  • ada-results.html — full results page: card design, data structure, nudge block
  • ada-claude-code-prompt.html — this spec document

Port copy, layout, and visual design 1:1 from the reference HTML files. Do not invent new copy or change the visual design — the reference files are the source of truth.

This is a portfolio piece. There is no backend, no authentication, and no database. Everything runs on static mock data. The goal is a fully interactive prototype a recruiter can click through without hitting dead ends.

Stack: React + TypeScript + Tailwind CSS + React Router, built with Vite. Deploy to Vercel via GitHub. All design tokens are already defined in the reference HTML — extract them into a Tailwind config or CSS variables file before building any component.

Database size

65 stays — exactly 5 per city across all 13 cities: Tokyo, Paris, New York, Lisbon, Tulum, Marrakech, Reykjavik, Kyoto, Florence, Cape Town, Mexico City, Bali, Barcelona.

Each city's 5 stays must span at least 3 different styles. Vary amenities and booking tags so every advanced filter produces a different result set. Calibrate prices to local market rates.

Data distribution rules

To ensure every filter toggle visibly changes results, enforce these minimums per city:

  • At least 2 stays with pool, at least 2 without
  • At least 2 stays with wifiSpeed >= 250, at least 1 under 100
  • At least 3 stays with cancellation, at least 1 without
  • At least 2 stays with reviews >= 100
  • Ratings spread between 4.70 and 4.98
  • At least 2 styles with maxGuests >= 4 and at least 1 with maxGuests <= 2

Picker function

export function pickStays(params: ResultsParams): Stay[] {
  let pool = [...STAYS]

  // 1. City — filter or shuffle for flexible
  if (params.where === 'flexible') {
    pool = shuffle(pool) // time-seeded, different every load
  } else if (params.where) {
    pool = pool.filter(s => s.city === params.where)
  }

  // 2. Guests — hard cut
  if (params.guests) {
    pool = pool.filter(s => s.maxGuests >= params.guests)
  }

  // 3. Advanced hard filters
  if (params.wifi && params.wifi !== 'any')
    pool = pool.filter(s => s.wifiSpeed >= Number(params.wifi))
  if (params.reviews && params.reviews !== 'any')
    pool = pool.filter(s => s.reviews >= Number(params.reviews))
  if (params.rating && params.rating !== 'any')
    pool = pool.filter(s => s.rating >= Number(params.rating))
  if (params.amenities?.length)
    pool = pool.filter(s =>
      params.amenities.every(a => s.amenities.includes(a)))
  if (params.booking?.length)
    pool = pool.filter(s =>
      params.booking.every(b => s.booking.includes(b)))

  // 4. Style soft sort — matching style floats to top
  pool.sort((a, b) => {
    if (params.style) {
      const aM = a.style === params.style ? 1 : 0
      const bM = b.style === params.style ? 1 : 0
      if (bM !== aM) return bM - aM
    }
    return b.rating - a.rating
  })

  // 5. Fallback — never fewer than 5
  if (pool.length < 5) {
    const base = STAYS
      .filter(s => !params.where || params.where === 'flexible'
                   || s.city === params.where)
      .sort((a, b) => b.rating - a.rating)
    pool = [...pool, ...base.filter(s => !pool.includes(s))]
  }

  return pool.slice(0, 5)
}

Context line builder

export function buildContextLine(params: ResultsParams): string {
  if (params.via === 'chat' || params.via === 'mood') {
    return params.q ?? ''  // shown italic with ochre left border
  }

  const parts: string[] = []

  if (params.style) parts.push(STYLE_LABELS[params.style] + ' stays')
  if (params.where === 'flexible') parts.push("Ada's flexible picks")
  else if (params.where) parts.push('in ' + params.where)

  if (params.checkin && params.checkout)
    parts.push(formatDateRange(params.checkin, params.checkout))

  const guests = params.guests ?? 0
  if (guests > 0) parts.push(guests + (guests === 1 ? ' guest' : ' guests'))

  const adv: string[] = []
  if (params.wifi && params.wifi !== 'any')
    adv.push(params.wifi + '+ Mbps Wi-Fi')
  if (params.booking?.includes('cancellation'))
    adv.push('free cancellation')
  if (params.booking?.includes('instant'))
    adv.push('instant book')
  if (adv.length) parts.push(adv.join(', '))

  return parts.join(' · ')
}

"I'm flexible" behaviour

When the user selects "I'm flexible" on the WHERE filter, pass where=flexible in the URL. On the results page, skip city filtering and shuffle the full 65-stay pool using a time-based seed so each page load produces genuinely different results. The context line should read: "Ada's flexible picks · [dates if present] · [guests if present]".

User states

Landing page — one version only: The landing page has identical copy and layout for both new and returning visitors. Do not build conditional rendering or alternate states on the landing page. The only place user state affects the UI is on the results page: new visitors see the nudge block, returning users don't.

There are two user states. For this portfolio build, implement them as a simple boolean — no real auth needed.

  • New visitor / no account — show the nudge block on the results page ("Ada gets better every time you use it"). Match badge says "Ada's pick #1–5".
  • Returning user / logged in — hide the nudge block. Match badge can show "96% match", "94% match" etc. (future state — wire the conditional now even if both states show the same data).
For now: default all visitors to the new-visitor state. Just wire the isLoggedIn boolean so it's easy to toggle later when auth is added.
